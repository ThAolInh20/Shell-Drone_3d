# Concerns & Risks

## 1. Performance & Particle Count
**The Threat:** Rendering thousands of particles directly via individual `THREE.Mesh` instances with unique materials can decimate framerates, especially on low-powered machines or mobile devices.  
**Mitigation Need:** Must switch to using `THREE.InstancedMesh` or `THREE.Points` (with custom Shaders if necessary) to batch draw calls for the `Star` entities instead of naive separate Geometry/Material per particle.

## 2. Unfinished Architecture Transition
**The Threat:** The `src/` directory defines a highly structured setup (`core`, `systems`, etc.), but `main.js` currently sidesteps it with inline boilerplate. 
**Mitigation Need:** The immediate technical debt relies on gutting `main.js` and properly wiring it up to consume the `Manager` controllers and the `System` lifecycle loop before starting on the complex business logic of fireworks.

## 3. Physics Frame Delta and Timesteps
**The Threat:** Utilizing hardcoded velocity values directly in `requestAnimationFrame` strings logic to framerate. A 144Hz monitor will behave differently than a 60Hz one.
**Mitigation Need:** Ensure that all velocity and physics (`GRAVITY`, `drag`) are rigidly multiplied against the `deltaTime` generated by `Clock.js` during the update step to guarantee deterministic or at least consistent simulation speeds.

## 4. Memory Leaks
**The Threat:** Fireworks mean high object churn. Constantly spawning and destroying thousands of objects leads to rapid Garbage Collection pauses. 
**Mitigation Need:** Implement object pooling. When a `Star` dies, do not destroy it (and its Three.js footprint). Instead, hide it and place it in a pool. Reuse the pool space when new fireworks burst. This maintains a flat memory profile.
